# 🔍 Phase 6: 분리된 평가자 (Harness Evaluator)

## 핵심 원칙 (Anthropic 하네스 논문 기반)
> "AI는 자기 작업을 관대하게 칭찬하는 경향이 있다.
>  평가는 반드시 구현과 분리된 관점에서 수행해야 한다."

> ⚠️ 이 워크플로우는 구현 완료 후 **별도로** 실행해야 한다. 구현한 AI가 스스로 평가하지 마라.

## 평가 트리거
- 매 Task의 구현(Phase 5) 완료 후
- 매 Epic 완료 시 전체 평가

---

## 프론트엔드 평가 기준 (4가지)

### 1. 디자인 품질 (가중치: 30%)
- 색상, 타이포, 레이아웃이 `docs/design-direction.md`와 일치하는가?
- 전체가 하나의 일관된 무드를 형성하는가?
- shadcn 컴포넌트의 스타일이 통일되어 있는가?
- 기준 미달 신호: 화면마다 다른 느낌, 색상 불일치, 간격 불규칙

### 2. 독창성 (가중치: 30%)
"AI 슬롭" 징후가 없는가?
- ❌ 보라색 그라데이션 위의 흰 카드
- ❌ shadcn 기본 테마 무수정
- ❌ 그라데이션 히어로 섹션 + 둥근 카드 3열
- ❌ 모든 페이지가 동일한 레이아웃
- 의도적인 디자인 결정이 보이는가?
- `docs/design-direction.md`의 무드 키워드가 반영되었는가?

### 3. 완성도 (가중치: 20%)
- 타이포 위계가 명확한가? (최대 3단계)
- 간격이 Tailwind spacing scale을 따르는가?
- 반응형이 깨지지 않는가? (모바일/태블릿/데스크톱)
- 로딩/에러/빈 상태가 모두 처리되어 있는가?

### 4. 기능성 (가중치: 20%)
- 주요 액션을 3초 내에 찾을 수 있는가?
- 사용자 흐름이 직관적인가?
- 폼 유효성 검사가 실시간으로 작동하는가?
- 키보드 네비게이션이 되는가?

---

## 백엔드/기능 평가 기준

### 1. 스펙 충실도
스프린트 계약(`tasks/contracts/TASK-XXX-contract.md`)의 완료 기준 대비:
- 각 기준을 PASS/FAIL로 판정
- FAIL에는 구체적 이유와 수정 지시 포함

### 2. 엣지 케이스
- 빈 입력 제출 시 적절한 에러?
- 네트워크 오류 시 사용자 피드백?
- 동시 요청 시 데이터 정합성?

### 3. 보안 (/security-audit 연동)
- 사용자 입력 검증 (Zod)?
- 인증/인가 체크?
- 환경변수 하드코딩?

## 평가 결과 출력 형식

```markdown
# 평가 보고서: TASK-XXX

## 종합 판정: PASS / FAIL

## 프론트엔드 점수
| 기준 | 점수(100) | 최소 기준(70) | 판정 |
|------|----------|-------------|------|
| 디자인 품질 | XX | 70 | ✅/❌ |
| 독창성 | XX | 70 | ✅/❌ |
| 완성도 | XX | 70 | ✅/❌ |
| 기능성 | XX | 70 | ✅/❌ |

## 스프린트 계약 충족
| # | 완료 기준 | 판정 | 비고 |
|---|----------|------|------|
| 1 | [기준] | PASS/FAIL | [구체적 이유] |

## FAIL 항목 수정 지시
[FAIL인 항목에 대한 구체적 수정 방향]

## AI 슬롭 감지
[해당 시 구체적 지적]
```

보고서 저장: `logs/phase-6-evaluation/eval-[번호].md`

---

## FAIL 시 흐름
FAIL → 구체적 피드백과 함께 `/prd-loop`의 Phase 3(구현)으로 반환
최대 3회 반복 후에도 FAIL → 사용자에게 보고 + 판단 요청

## 금지 사항
- 자기 작업을 평가하지 마라 (분리 원칙).
- "전반적으로 잘 되었습니다"같은 모호한 평가 금지.
- 70점 미만인 항목이 있으면 반드시 FAIL 판정.
