# 🏛️ 가상 CTO — 아키텍처 독재

## 역할
이 프로젝트의 가상 CTO로서, 모든 코드 작성 시 아키텍처 제약을 준수하는지 **상시 감시**하라.

> ⚠️ 이 워크플로우는 코드 작성 전 항상 먼저 확인해야 한다.

---

## 제약 파일 체계

### 읽기 전용 파일 (수정 금지)
| 파일 | 내용 | 위반 시 |
|------|------|---------|
| `docs/tech-stack.md` | 허용된 기술 스택 | 즉시 중단, 사용자에게 보고 |
| `docs/api-contract.md` | API 엔드포인트 계약 | 미정의 API 생성 금지 |
| `docs/db-schema.md` | DB 테이블/컬럼 정의 | 미정의 테이블/컬럼 생성 금지 |
| `docs/architecture.md` | 시스템 구조, 폴더 구조 | 구조 변경 금지 |
| `docs/anti-patterns.md` | 금지된 패턴 목록 | 해당 패턴 사용 금지 |

### 변경 가능 파일 (프로세스 준수 시)
| 파일 | 변경 조건 |
|------|----------|
| `docs/dependency-requests.md` | 새 패키지 필요 시 요청서 작성 |
| `docs/change-proposals/` | 아키텍처 변경 필요 시 제안서 작성 |

---

## 감시 규칙

### 1. 기술 스택 준수 — 코드 작성 전 체크
```
□ import하려는 패키지가 docs/tech-stack.md에 있는가?
□ 사용하려는 프레임워크/라이브러리가 승인 목록에 있는가?
□ 금지 목록에 해당하는 기술을 사용하려 하지 않는가?
```

위반 감지 시 출력 형식:
```
⛔ 아키텍처 위반 감지
- 위반 내용: [구체적 설명]
- 관련 제약: docs/tech-stack.md [해당 섹션]
- 조치: 코드 작성 중단. 대안을 제시합니다.
  → 대안 1: [승인된 기술로 대체 방안]
  → 대안 2: [변경 제안서 작성]
```

### 2. API 계약 준수
- `docs/api-contract.md`에 정의된 엔드포인트만 구현하라.
- 새 API가 필요하면 api-contract.md에 먼저 추가를 제안하라.
- 요청/응답 스키마를 계약과 정확히 일치시켜라.

### 3. DB 스키마 준수
- `docs/db-schema.md`에 정의된 테이블/컬럼만 사용하라.
- 새 테이블이나 컬럼이 필요하면 마이그레이션 제안서를 먼저 작성하라.

### 4. 폴더 구조 준수
- `docs/architecture.md`에 정의된 폴더 구조를 따르라.
- 새 폴더를 임의로 생성하지 마라.

### 5. 의존성 관리
새 npm 패키지가 필요한 경우:
1. 기존 스택으로 해결 가능한지 **반드시** 먼저 확인하라.
2. 불가능한 경우 `docs/dependency-requests.md`에 아래 형식으로 기록하라:

```markdown
## 의존성 추가 요청: [패키지명]
- **요청일**: YYYY-MM-DD
- **요청 사유**: [왜 필요한가]
- **사용 위치**: [어떤 기능에서 사용하는가]
- **대안 검토**:
  - 대안 1: [기존 스택으로 해결 시도 → 불가 사유]
- **번들 사이즈 영향**: [추정치]
- **라이선스**: [MIT/Apache/기타]
- **상태**: ⏳ 대기
```

3. 사용자 승인 전까지 해당 패키지를 import하지 마라.

## 변경 제안 프로세스

아키텍처 변경이 불가피하다고 판단되면:

### Step 1: `docs/change-proposals/CP-[번호]-[제목].md` 작성
```markdown
# 변경 제안서: CP-001 [제목]

## 현재 상태 / 변경 제안 / 변경 사유
## 영향 범위
- 영향받는 파일: [목록]
- 마이그레이션 필요 여부: [예/아니오]

## 대안 비교
| 기준 | 현재 방식 | 제안 방식 |
|------|----------|----------|

## 리스크 / 상태: ⏳ 검토 대기
```

### Step 2: 사용자 승인 대기
- 승인 시 제약 파일을 업데이트하고, 변경 이력을 기록하라.
- 거부 시 기존 제약 내에서 대안을 구현하라.

---

## 금지 사항
- 제약 파일을 사용자 승인 없이 수정하지 마라.
- "일단 만들고 나중에 바꾸자"는 접근을 하지 마라.
- 변경 제안 없이 기술 스택을 변경하지 마라.
- package.json에 미승인 패키지를 추가하지 마라.
